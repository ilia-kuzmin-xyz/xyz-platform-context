## Mechanism (code-verified) — how the new-dashboard Actual % is produced

All refs in `hc-frontend/src/main/webapp/app/pages/organisation/ViewerPage/`:

1. **Per-activity progress is pre-computed on the backend and read verbatim.** `activity-progress-v2-loader.ts:60-155` loads the `planned-and-actual-activity` parquet from `/api/v2/projects/{projectId}/progress-outputs` as a zero-copy VIEW `activity_progress` with columns `ActivityId, CalendarDate, PlannedProgress, ActualProgress, PlannedLaborUnits, LinkedElements` — **all pre-calculated 0–1 values** (`:62-68`). The dashboard never recomputes actual from labour or elements.

2. **The FE does NOT implement the intangible fallback.** The active V2 aggregation (`utils/progress-queries-v2-api.ts`) sums `${weightColumn} * ${actualProgressCol}` straight from the parquet columns — project level (`:124-152`), package level (`:193-263`), category summary (`:563-590`), activity level (`:687-736`). There is **no `LinkedElements = 0 → ActualLaborUnits/PlannedLaborUnits` branch anywhere in the query layer**; `ActualLaborUnits` exists only as a raw row/debug type (`utils/progress-data.types.ts:69`, `types.ts:141`), not in any aggregation. **Consequence: if the parquet says `ActualProgress = 0` for an intangible activity, the new dashboard shows 0 for it, and no front-end change can alter that** — the fix would be in the backend parquet generator (the skill's "upstream parquet generator caveat").

3. **Activity progress is scoped to one schedule revision.** `activity-progress-v2-loader.ts:113-114` bakes `WHERE ScheduleRevisionId = '${this.currentScheduleRevisionId}'` into the VIEW. `category_groups` / `project_progress` carry `ScheduleRevisionId` too (`progress-queries-v2-api.ts:35,54`). **Consequence: only activities present in the current schedule revision contribute.** An activity absent from the uploaded XER/revision has no rows and silently drops out of both numerator and denominator.

4. **Weighted aggregation excludes zero-weight rows.** Every category query filters `AND ${weightColumn} > 0` (`:218,235,445,577`) and the category list drops all-zero rows (`:599-603`). So an intangible activity/package with `TotalLinkedElements = 0` (element mode) or `TotalPlannedLaborUnits = 0` contributes nothing — by design.

**Both H1 and H2 therefore produce exactly the reported shape** (new Actual < old Actual; named activities showing 0 on the new dashboard). The code cannot, by itself, distinguish which one is happening for `EL1031000` — that requires looking at the data (does EL1031000 exist in the XER, and if so what are its `LinkedElements` / `PlannedLaborUnits` / `ActualProgress` in the parquet).

### Why the customer's "editor shows the progress" note matters
If `EL1031000` genuinely shows progress in the **web viewer Editor-Progress**, the activity **exists in the platform's activity/schedule data** — which sits awkwardly with "missing from the source data / it's the XER file" (H1) and leans toward **H2 (present but parquet actual = 0)**. Caveat: the editor and the dashboard may read **different schedule revisions**, and "Editor-Progress" may reflect element installation status rather than the same aggregated activity row — so this is suggestive, not conclusive. It is exactly the tension a data check resolves.

### "Install Temp Power" is a textbook intangible activity
Temporary-power installation is not typically modelled BIM geometry → likely `LinkedElements = 0` → the backend uses the **intangible (labour-based) actual path** — the exact path Pattern A breaks. This makes H2 a live, not exotic, hypothesis for this specific activity.
