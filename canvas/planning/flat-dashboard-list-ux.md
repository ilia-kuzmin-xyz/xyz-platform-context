# Flat Dashboard List ŌĆö UX Concept

> **Status: CONCEPT ŌĆö needs UX design review before implementation.**
> Do NOT implement until concept is reviewed and approved.

## Problem

Currently the only way to reach a past dashboard is:
1. Open the specific session it belongs to
2. Scroll the chat to find that turn's "View dashboard" button

If a user remembers "I built a great issues dashboard last week" but doesn't remember which session it was in, they have to open sessions one by one and scroll each chat. There's no cross-session dashboard discovery.

---

## What we want

A **flat list of all dashboards across all sessions** for the current project. User can browse, search, and directly open any past dashboard without knowing which session it came from.

---

## Data available (no new storage needed)

Each session file already contains `dashboards[]`:
```json
{
  "id": "d2",
  "messageId": "m2",
  "title": "Issues vs Progress",
  "summary": "Project is 60% aheadŌĆ”",
  "domainsRead": ["issues","progress"],
  "createdAt": 1730000005000
}
```

And the session gives us:
- `sessionId`, `sessionName`, `createdBy`, `updatedAt`

So we can build: `{ dashboardId, title, summary, domainsRead, createdAt, sessionId, sessionName }` for every dashboard across all sessions.

**Cost**: loading the full list requires downloading all session files (one `GET /files/{ref}` per session). For 10 sessions this is 10 requests + 10 downloads. Acceptable.

---

## UX Options

### Option A ŌĆö Second panel: "All Dashboards"

Alongside the existing `SessionsPanel`, add a toggle between two views:
```
[ Sessions ]  [ Dashboards ]
```
"Dashboards" shows a flat list sorted by `createdAt` desc. Clicking a dashboard opens its session and activates that dashboard.

**Pros**: clean separation; SessionsPanel UX unchanged.
**Cons**: another panel mode to maintain; toggling adds a click.

---

### Option B ŌĆö Expand sessions inline

In the `SessionsPanel`, each session row is expandable ŌĆö clicking the chevron reveals that session's dashboards inline.

```
Ō¢Č "API 2 weekly review"   Jun 3  ŌĆó  3 dashboards
Ō¢Č "Issues analysis"       Jun 1  ŌĆó  2 dashboards
```

Expanded:
```
Ō¢Š "API 2 weekly review"   Jun 3
    Ōöö Issues vs Progress    Jun 3  [Open]
    Ōöö Schedule Overview     Jun 3  [Open]
    Ōöö Progress S-Curve      Jun 2  [Open]
```

**Pros**: hierarchical (you see which session each dashboard came from); no new panel.
**Cons**: long lists can be hard to scan; requires all session files downloaded on expand.

---

### Option C ŌĆö Separate "Dashboards" entry in the main canvas chrome

Rather than inside the sessions panel, add a persistent "Dashboards" tab or button in the ArtifactPanel header (or the top bar). Opens a grid/list of dashboard cards with title + summary preview.

```
[ Dashboard: Issues vs Progress ]  [ Dashboard: S-Curve ]  [ + New session ]
```

**Pros**: always accessible; good for power users with many dashboards.
**Cons**: takes up real estate in the main canvas area; needs a good empty state.

---

### Option D ŌĆö Search / command palette

A search input (Cmd+K or within the sessions panel) that searches across all dashboard titles and summaries. Type "issues" ŌåÆ shows all dashboards mentioning issues.

**Pros**: fastest path for users who know what they want; scales to many dashboards.
**Cons**: requires loading all session files to build the index; complex to implement.

---

## Open questions

1. **Loading cost**: displaying the flat list requires downloading ALL session files to get dashboard arrays. For projects with 50+ sessions this could be slow. Should we lazy-load (expand on demand) or eager-load (download all on panel open)?

2. **Navigation**: when the user opens a dashboard from the flat list, do they:
   - **Land in the session** (chat visible, that dashboard active, "ŌåÉ Back to All Dashboards" button)?
   - **Just see the dashboard** (no chat, pure fullscreen artifact)?

3. **Session context**: does a "flat list" dashboard open need the session chat? Users might want to see the conversation context that produced the dashboard.

4. **Hydration on open**: clicking a dashboard from the flat list should trigger hydration. Same path as "View dashboard" button in chat.

5. **Naming**: dashboards currently have `title` (auto-generated by the pipeline). Should users be able to rename them? That requires saving the new title back to the session file.
