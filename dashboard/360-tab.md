# 360 Captures Tab

Room-based panoramic captures with spatial and temporal filtering.

## What the user sees

- **Room list** with thumbnails, capture count per room, and latest capture date.
- **Level metadata** (e.g. "LEVEL 01") shown per room.
- **Filtering** by date range, building level, and room name.
- **Multi-select** support for rooms.
- **Detail panel** opens on room selection with image viewer.
- **Capture point names** when available.

## Data sources

| Source | What it provides |
|--------|-----------------|
| 360 Captures API (v2) | Panoramic capture records (paginated, 1000 per page) |
| Room Capture API | Room metadata and capture point names |
| DuckDB tables | `captures_360` and `rooms_360` for SQL filtering |

## How filtering works

Filters are applied at the **capture level** first, then results are grouped by room:

```sql
SELECT ... FROM captures_360
WHERE imageTakenOn >= :startDate
  AND imageTakenOn <= :endDate
  AND levelName IN (:selectedLevels)
GROUP BY modelRoomId
```

Rooms without matching captures disappear from the list. Capture counts reflect the filtered subset, not the total. This ensures users only see relevant rooms.

## Lazy initialization

Like Quality, the 360 service initializes only when the tab is opened. However, room/level **metadata** is loaded eagerly so the filter panel is populated before the user opens the tab.

## Implementation notes

- Thumbnail URLs go through a CORS proxy via `getStorageUrl()`.
- Capture point names require a separate API call (cached in `_capturePointNames`).
- The room list currently reuses the `IssueTable` component from the Quality panel with adapted data — a pragmatic choice but somewhat awkward.
